# 決定の要約

CodeRabbit の指摘は「マージ後スクリーニング標準化」（ベストエフォート + 後追い）で扱う。マージは CodeRabbit 完了を待たない。判定結果を #45 に蓄積して的中率を計測し、データが溜まった時点で「高確信チェックのみ gate 昇格」を再検討する。それまで gate は tidd ai-review 単独を維持する。

- **決定日:** 2026-07-21
- **記録者:** Claude（ユーザー指示に基づく）
- **参照:** tan3159/trade-scripts#27 #44 #45 / being-gaia-plan/ai-dev-handbook#2340 / docs/decisions/2026-07-19-public-repo-improvements-upstream-first.md

## 論点

CodeRabbit（advisory・#27 で有効化）のレビュー完了は tidd ai-review のマージより遅れることがあり、1 Issue 1 セッション運用ではマージ後の指摘を見返すトリガーがない。待ち時間を増やさず取りこぼしをなくす方法と、二重 gate 化（双方 APPROVE でマージ）の是非。

## 提示した選択肢

| 案 | 内容 |
|----|------|
| A | マージ後スクリーニング標準化 — マージは待たせず、マージ後クリーンアップで CodeRabbit 完了をポーリング → 指摘を分類 → 妥当分のみ起票（推奨） |
| B | 上限付き待ち（マージ前に最大 3 分待つ） |
| C | 完全待ち（CodeRabbit 完了までマージしない） |

gate 化については: 業界の主流は advisory 運用で、gate 化する場合も「少数の高確信ルールのみ昇格」が定石。導入初期は的中率を追跡し誤検知率で判断するのがベストプラクティス。

## ユーザーの決定

**A を採用。** さらに「A の運用で的中率データを蓄積 → データが溜まったら高確信チェックのみ gate 昇格を検討」の道筋も承認。

## 理由・背景

- 指摘コメントは PR に残るため、失うのは「見返す機会」だけ。マージ後スクリーニングの標準化で構造的にゼロにできる
- advisory のためにマージを遅らせる理由がない（失うのは同一 PR 内修正の機会のみ・別 Issue で回収可能）
- スクリーニング記録がそのまま gate 昇格判断の実データになる（業界ベストプラクティスと整合）
- 「advisory 単独 gate」は 2026-07-19 決定だが、ユーザーは推奨に乗っただけであり、データに基づく再検討はオープン

## 今後 AI が取るべき行動

1. 各 PR マージ後のクリーンアップで CodeRabbit 指摘をスクリーニングし、判定結果（PR 番号・指摘数・妥当/誤検知/スコープ外・起票 Issue）を #45 にコメント記録する（上流 #2340 の仕組みが届くまでは手動実施）
2. 妥当かつ Issue スコープ外の指摘は issue-creation ルール準拠で起票する
3. マージ前に CodeRabbit 完了を待たない（B/C 方式を提案しない）
4. #45 の条件（PR 20 件 or 2026-09-30）到達時に的中率を集計し、gate 昇格の選択肢をエスカレーションする。それまで二重 gate 化は提案しない
